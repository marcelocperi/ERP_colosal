Manual General del Sistema MultiMCP
Versión 2.0 - Documentación Integral

Este manual describe la operatividad completa del sistema, abarcando infraestructura, gestión multi-empresa y módulos operativos. Diseñado para usuarios clave y auditores.

1. Infraestructura y Arquitectura
El sistema opera bajo un modelo Multi-Tenant (Múltiples empresas en una sola instalación), garantizando el aislamiento de datos y la seguridad.


2. Gestión de Empresas
Módulo administrativo para la creación y configuración de nuevos entornos de trabajo.


3. Gestión de Stock e Inventario
Control total de los bienes de cambio, desde su definición hasta su movimiento físico.


4. Módulo de Compras
Ciclo completo de aprovisionamiento con controles de segregación de funciones.


5. Módulo de Ventas
Gestión comercial desde la cotización hasta la facturación.


6. Tesorería y Fondos
Administración centralizada del flujo de dinero (Cash Flow).

Para que la migración asincrónica de Georef sea completa y el sistema mantenga una arquitectura coherente, los lugares clave que faltan migrar son:
1. Endpoints Auxiliares en 

core/routes.py
Actualmente, solo convertimos /api/georef/calles. Para mantener la consistencia y evitar bloqueos si el sistema necesita escalar, deberíamos migrar los siguientes:
/api/georef/localidades
/api/georef/municipios
/api/georef/municipio_info
Razón: Aunque hoy consultan la base de datos local (que es rápida), migrarlos a async def garantiza que cualquier lógica futura de fallback a la API externa sea segura.
2. Métodos de Sincronización en 

GeorefService
Los métodos que descargan los datos maestros (

load_provincias, 

load_municipios, 

load_localidades) siguen usando requests.get (sincrónico). Si decidimos agregar un botón de "Actualizar Geografía" en el Panel de Administración, estos métodos congelarían el servidor durante varios segundos debido al volumen de datos.
3. Otros Módulos con Selectores Geográficos
Debemos verificar que las llamadas asincrónicas funcionen correctamente en las siguientes pantallas que ya utilizan estos selects dinámicos:
Compras: Ficha de Proveedores (

perfil_proveedor.html).
Stock: Alta de Depósitos / Sucursales (

deposito_form.html).
Logística: Scripts y rutas de normalización de direcciones para entregas.
4. AFIP: El pendiente crítico
Fuera de Georef, en el servicio de AFIP realizamos la migración de la consulta de Padrón, pero aún falta migrar el método 

solicitar_cae. Este es el corazón de la facturación electrónica; migrarlo permitirá que el servidor no se bloquee mientras AFIP procesa y autoriza facturas legales.
